1. DHCP Starvation

Текущая реализация:

    Отправка DHCP Discover и DHCP Request с уникальными MAC-адресами.

    Параметры: интерфейс, размер пула (количество уникальных MAC), количество запросов, задержка.

    Статистика: количество отправленных пакетов, уникальных MAC, время работы, скорость.

Предложения по улучшению:

    Реалистичность: добавить поддержку опций DHCP (например, параметры запроса, vendor class), чтобы имитировать разные типы клиентов (Windows, Linux, мобильные). Это поможет обойти простые фильтры.

    Ускорение: использовать многопоточность для отправки запросов параллельно (например, 4-8 потоков).

    Режимы: реализовать «агрессивный» режим, где Discover и Request отправляются одновременно разными потоками, или «медленный» для скрытности.

    Случайность XID: уже есть, но можно добавить опцию использования фиксированного XID для тестирования.

    Обработка ответов: можно добавить сниффинг DHCP-ответов (offer/ack) для подсчёта реально захваченных адресов и подтверждения успешности атаки.

    Интеграция с ARP spoofing: после истощения пула можно автоматически включить ARP spoofing для перехвата трафика.

    Логирование: выводить список захваченных IP (если удалось получить offer) для последующего анализа.

Применение:
Тестирование устойчивости DHCP-серверов, проверка наличия защиты (DHCP snooping, rate limiting), имитация клиентов для заполнения таблиц коммутатора.

Ожидаемый результат:
При успешной атаке пул адресов исчерпывается, новые клиенты не могут получить IP. Включение DHCP snooping может заблокировать атаку, но если snooping настроен неправильно или отключён, атака будет эффективна.
2. DoS атаки (UDP, ICMP, TCP, ARP, DNS)
2.1 UDP flood (через raw-сокеты)

Текущая реализация:

    Многопоточная отправка UDP-пакетов через raw-сокеты с ручным формированием IP и UDP заголовков.

    Кеширование данных и псевдозаголовков для ускорения.

    Параметры: целевой IP, порт, размер пакета, время (или бесконечно), интерфейс, source IP (определяется автоматически).

Предложения по улучшению:

    Динамический source IP: уже есть фиксированный source IP интерфейса. Можно добавить опцию случайного source IP (как в ICMP/TCP/ARP) для имитации распределённой атаки.

    Размер пакета: ограничен MTU? Нужно учитывать, что IP-фрагментация может вызвать дополнительные проблемы на цели, но raw-сокеты обычно позволяют отправлять пакеты больше MTU, и они будут фрагментироваться стеком. Лучше дать возможность задать размер с учётом фрагментации или отправлять нефрагментированные Jumbo-фреймы.

    Тип UDP-полезной нагрузки: можно сделать выбор: случайные данные, фиксированный текст, нули.

    Случайные порты: уже реализовано варьирование source port.

    Статистика: отображается PPS, общее количество, байты. Хорошо бы добавить график скорости в реальном времени.

    Интеграция с внешними утилитами: если raw-сокеты не работают (например, на некоторых системах), можно использовать hping3 или собственный .exe.

2.2 ICMP flood (через внешний icmp.exe)

Текущая реализация:

    Запуск внешнего icmp.exe с параметрами: source IP, target IP, число потоков, время, опции random IP/MAC.

    Обработка JSON-статистики от внешнего процесса.

Предложения по улучшению:

    Интеграция в основной код: если возможно, реализовать ICMP flood через Scapy или raw-сокеты, чтобы не зависеть от внешнего exe.

    Типы ICMP: не только echo, но и timestamp, info request, address mask request – некоторые старые системы отвечают на них, увеличивая нагрузку.

    Размер пакета: добавить возможность настраивать размер ICMP payload.

    Случайный source IP: уже есть, но для реальной эффективности лучше использовать подсеть жертвы (чтобы ответы возвращались к ней) или случайные IP из публичного диапазона.

    Опция использовать directed broadcast для усиления (Smurf-like, но с подменой source IP жертвы).

2.3 TCP flood (через внешний tcp.exe)

Текущая реализация:

    Внешний tcp.exe с параметрами: source IP, target IP, порт, потоки, время, random IP/MAC.

Предложения по улучшению:

    Типы TCP-пакетов: сейчас, вероятно, только SYN (полуоткрытые соединения). Добавить поддержку других флагов: SYN-ACK, RST, ACK, FIN, а также комбинации (XMAS, NULL) для обхода фильтров.

    Опции TCP: возможность задавать window size, sequence number, опции (MSS, window scale, SACK) для имитации реального стека.

    Случайный port назначения: можно добавить режим «port scan» – перебор портов.

    Учёт состояния: если отправлять только SYN, можно отслеживать количество SYN-ACK ответов для оценки успешности (но это сложнее).

2.4 ARP flood (через внешний arp.exe)

Текущая реализация:

    Внешний arp.exe с параметрами: source IP, target IP, потоки, время, random IP/MAC.

Предложения по улучшению:

    Типы ARP-пакетов: сейчас, вероятно, ARP-запросы (who-has). Можно добавить ARP-ответы (is-at) для отравления кэша.

    Скорость: ARP – широковещательный, может загрузить коммутатор. Улучшить можно многопоточностью и увеличением частоты.

    Подмена MAC: случайный MAC уже есть. Хорошо бы дать возможность задавать диапазон OUI для имитации разных вендоров.

    Интеграция с ARP spoofing: после ARP flood можно автоматически включить ARP spoofing для перехвата.

2.5 DNS flood (через Scapy)

Текущая реализация:

    Отправка DNS-запросов (тип A) на целевой DNS-сервер с разными случайными доменами.

    Многопоточность (число потоков = min(4, cpu_count)).

    Параметры: target IP, время, непрерывно, интерфейс.

Предложения по улучшению:

    Типы запросов: добавить поддержку других типов (AAAA, MX, TXT, NS) для разнообразия.

    Реалистичные домены: можно использовать список популярных доменов или генерировать случайные имена, которые с большей вероятностью потребуют рекурсивного разрешения (например, случайные поддомены существующих доменов).

    EDNS0: добавить поддержку расширений (больший размер UDP) для увеличения нагрузки.

    TCP-режим: при превышении размера ответа DNS может переключаться на TCP. Можно отправлять запросы, специально урезанные, чтобы вызывать TCP-соединения.

    Усиление (amplification): если DNS-сервер открытый, можно подделывать source IP жертвы и отправлять запросы с малым payload, получая большой ответ (амплификация). Это классическая DRDoS. Реализовать опцию "amplification" с подменой source IP (если сеть позволяет).

    Статистика: отображать не только количество пакетов, но и количество ответов (если сниффить), чтобы оценить эффективность.

Общие улучшения для всех DoS атак:

    Единый интерфейс управления: сделать общую панель статистики с графиками PPS и Mbps (можно через matplotlib в отдельном окне).

    Профили атак: возможность сохранять и загружать конфигурации.

    Планировщик: запуск атаки по расписанию или при обнаружении события (например, при перехвате определённого пакета).

    Тестирование защит: добавить возможность использовать случайные интервалы между пакетами (jitter) для имитации реального трафика.

    Поддержка IPv6: все протоколы должны иметь опцию работы с IPv6.

3. ARP Spoofing

Текущая реализация:

    Периодическая отправка поддельных ARP-ответов цели и шлюзу с MAC-адресом атакующего.

    Интервал задаётся пользователем.

    Статистика: отправлено пакетов, время работы, скорость.

Предложения по улучшению:

    Двунаправленный спуфинг: уже есть (и цель, и шлюз), хорошо.

    Восстановление ARP: после остановки можно автоматически отправить правильные ARP-ответы для восстановления нормальной работы (опционально).

    Перехват трафика: добавить функцию включения IP forwarding (в Windows нужно включить через реестр или использовать готовые утилиты) и, возможно, простой форвардинг с логированием.

    Фильтрация перехваченного трафика: можно добавить возможность перехватывать только определённые порты (например, HTTP) и сохранять в pcap.

    Интеграция с перехватом пакетов: после успешного спуфинга автоматически запускать перехват трафика между целью и шлюзом.

    Поддержка нескольких целей: дать возможность указывать список IP для отравления (например, все хосты в подсети).

    Обнаружение: добавить опцию "обнаружение ARP spoofing" – прослушивать ARP-ответы и выявлять несоответствия.

Применение:
MITM-атаки, тестирование IDS/IPS, проверка корректности настройки ARP inspection на коммутаторах (DAI).

Ожидаемый результат:
Трафик между целью и шлюзом проходит через атакующего (при включённом IP forwarding). При отсутствии защиты – полный перехват. DAI (Dynamic ARP Inspection) может заблокировать атаку, если порт не trusted.
4. VLAN Flood

Текущая реализация:

    Запуск внешнего vlan.exe с параметрами: интерфейс, VLAN ID, число потоков, время, опции random MAC или фиксированный MAC.

    Отображение статистики через JSON от внешнего процесса.

Предложения по улучшению:

    Интеграция в основной код: если возможно, реализовать через Scapy с многопоточностью и статистикой, чтобы не зависеть от внешнего exe.

    Диапазон VLAN: добавить возможность указывать диапазон VLAN ID для перебора (например, 1-4094) или список.

    Payload: дать возможность настраивать полезную нагрузку (текст, случайные данные, определённый протокол внутри – IP, ARP, Raw).

    Опции MAC: уже есть random и fixed. Хорошо бы добавить возможность использования MAC интерфейса (auto).

    Учёт 802.1ad (QinQ): можно добавить поддержку двойного тегирования (как в VLAN Hop), но для flood – отправлять кадры с разными комбинациями внешних/внутренних тегов.

    Статистика: отображать количество отправленных кадров, количество уникальных VLAN, скорость (fps). Сейчас это делает внешний процесс через JSON, но можно унифицировать.

Применение:
Тестирование коммутаторов на перегрузку при обработке большого числа VLAN, проверка фильтрации нетегированного трафика, DoS на VLAN-интерфейсы.

Ожидаемый результат:
При интенсивной отправке кадров с разными тегами может возрасти нагрузка на CPU коммутатора, переполниться таблица VLAN, или, если коммутатор неправильно настроен, возможно проникновение в другой VLAN (если используется двойное тегирование).
5. Packet Intercept (сниффер + ответы)

Текущая реализация:

    Захват пакетов на интерфейсе с фильтром BPF.

    Автоматическая отправка заданного количества ответных пакетов (ICMP Echo Reply на Ping, TCP RST/ACK, UDP-ответ).

    Возможность захватить выбранный пакет, отредактировать и отправить его.

    Таблица перехваченных пакетов с деталями.

Предложения по улучшению:

    Улучшение генерации ответов:

        Для TCP: вместо RST/ACK можно отправлять SYN-ACK для установки соединения (полуоткрытое), а затем FIN или RST.

        Для UDP: можно имитировать ответ от реального сервиса (например, DNS-ответ, HTTP-ответ).

        Дать пользователю возможность задать шаблон ответа (например, через поле "Payload" или выбор из предустановленных).

    Сохранение в pcap: добавить кнопку для сохранения всех перехваченных пакетов в файл .pcap.

    Фильтрация по протоколу/порту: уже есть BPF, но можно добавить простые галочки (TCP, UDP, ICMP, ARP) для быстрой установки фильтра.

    Повторная отправка: возможность отправить выбранный пакет многократно (как DoS) с заданной задержкой.

    Редактор пакетов: уже есть, но можно улучшить: добавить поддержку всех полей Scapy, предварительный просмотр изменений, возможность загрузки/сохранения пакета в hex.

    Статистика: показывать количество перехваченных пакетов в секунду.

    Интеграция с другими атаками: после перехвата определённого пакета (например, DHCP Discover) можно автоматически запустить атаку (например, DHCP starvation).

    Мониторинг: добавить возможность прослушивать трафик и выводить только определённые поля (например, HTTP-запросы).

Применение:
Анализ трафика, отладка сетевых приложений, создание собственных ответов для тестирования, изучение протоколов.

Ожидаемый результат:
Пользователь может видеть весь трафик в реальном времени, манипулировать им и отправлять модифицированные пакеты.
6. Access Tab (утилиты)

Текущая реализация:

    Ping (через subprocess), Port Scan (простейший по списку портов), Traceroute (через tracert/traceroute), show IP route (через route print), show network adapters (через get_if_list).

Предложения по улучшению:

    Ping:

        Добавить опции: количество пакетов, размер, таймаут, интервал.

        Реализовать через Scapy для более точного контроля (например, возможность подделки source IP).

    Port Scan:

        Улучшить: сканирование диапазона портов, выбор типа сканирования (TCP connect, SYN stealth, UDP, FIN, XMAS).

        Добавить определение сервисов (banner grabbing).

        Многопоточность для ускорения.

    Traceroute:

        Добавить выбор протокола (ICMP, UDP, TCP) и порта.

        Возможность задать максимальное количество хопов, таймаут.

        Отображать геолокацию IP (через внешний API, опционально).

    IP route:

        Улучшить парсинг вывода route print для Windows, чтобы выводить в виде таблицы.

        Добавить кнопку для сохранения в файл.

    Network adapters:

        Показывать не только интерфейсы Scapy, но и все сетевые адаптеры системы с их состоянием, IP, MAC, MTU.

        Возможность включить/выключить адаптер (требует прав администратора).

    Дополнительные утилиты:

        Whois – запрос whois по IP/домену.

        Nslookup/Dig – DNS-запросы.

        ARP-таблица – показать кэш ARP системы.

        Netstat – текущие соединения.

        Route add/delete – возможность добавлять/удалять маршруты (осторожно!).

        Wake-on-LAN – отправка магического пакета.

        Трафик-генератор – отправка произвольного пакета с заданными параметрами.

Применение:
Диагностика сети, сбор информации, базовое тестирование доступности.

Ожидаемый результат:
Удобный набор инструментов для сетевого администратора или пентестера прямо в одном окне.
7. Общие улучшения для всего приложения

    Профили и сохранение настроек: сохранять в JSON последние использованные параметры (интерфейс, IP, порты) для каждой вкладки, чтобы при следующем запуске не вводить заново.

    Горячие клавиши: добавить сочетания для запуска/остановки атак (например, F5 – старт, F6 – стоп).

    Уведомления: при завершении атаки по таймеру показывать всплывающее уведомление.

    Логирование в файл: все сообщения из логов можно автоматически сохранять в файл с датой.

    Проверка прав: уже есть проверка администратора, но также проверять наличие Npcap/WinPcap, иначе выводить предупреждение.

    Модульность: вынести атаки в отдельные классы (как уже сделано для RSattack и Sattack), чтобы код был чище.

    Документация: улучшить справку, добавить примеры использования каждой атаки.

    Перевод интерфейса: сделать возможность переключения языка (английский/русский), либо оставить как есть.

    Темы оформления: уже есть, можно добавить ещё несколько (например, "хай-тек", "милитари").
    Анализ и предложения по улучшению L2 и Network атак
Общие замечания

Все атаки в разделах L2 и Network уже реализованы базово, но имеют потенциал для доработки, чтобы стать более реалистичными, эффективными и удобными для тестирования защиты. Ниже приведены конкретные рекомендации для каждого типа атаки, а также общие предложения по интеграции и расширению функционала.
1. STP BPDU flood (L2)

Текущая реализация:
Отправка кадров с LLC (dsap=0x42, ssap=0x42) и сырыми данными \x00... на мультикаст-адрес 01:80:c2:00:00:00.
Пакет не содержит полноценных BPDU-полей, только заглушка.

Предложения по улучшению:

    Генерировать полноценные BPDU-пакеты (например, Configuration BPDU) с корректными полями:

        Protocol ID (0), Version (0), Type (0), Flags (0), Root ID, Path Cost, Bridge ID, Port ID, Message Age, Max Age, Hello Time, Forward Delay.

    Дать возможность выбирать тип BPDU (Config / TCN).

    Добавить параметры для Root Bridge ID, Bridge ID, Path Cost, чтобы имитировать разные сценарии (root guard bypass, topology change).

    Реализовать поддержку VLAN-тегов (802.1Q), так как STP работает per VLAN (MSTP, PVST+).

    Интегрировать счётчик отправленных пакетов и скорость (pps), как в других атаках.

Применение:
Тестирование устойчивости коммутаторов к поддельным BPDU, проверка работы защит (BPDU Guard, Root Guard, TCN Guard), создание временных петель для DoS.

Ожидаемый результат:
При отсутствии защит — пересчёт spanning tree, возможные петли, потеря связи. Современные сети с BPDU Guard на access-портах просто заблокируют порт, но flood большого числа BPDU может перегрузить CPU управляющих модулей.
2. CDP flood (L2)

Текущая реализация:
Кадры LLC/SNAP (OUI=0x00000C, code=0x2000) с сырым payload \x01\x00. Это лишь минимальный CDP-пакет.

Предложения по улучшению:

    Сформировать валидный CDP-пакет с TLV (тип-длина-значение):

        Device ID, Version, Platform, Addresses, Port ID, Capabilities, Software Version и т.д.

    Дать возможность изменять эти поля через GUI (например, задавать имя устройства, платформу).

    Увеличить длину пакета (можно добавлять множество повторяющихся TLV или padding) для усиления эффекта переполнения буферов.

    Добавить поддержку CDPv2.

Применение:
Атака на устройства Cisco (и некоторые другие, поддерживающие CDP) с целью заполнения таблиц соседей, повышения нагрузки на CPU, обхода аутентификации (CDP не аутентифицируется).

Ожидаемый результат:
На коммутаторах Cisco массовая отправка CDP может привести к исчерпанию памяти под таблицы CDP, сбою протокола, а в некоторых старых версиях IOS — к перезагрузке. На современных устройствах с CDP Guard или отключённым CDP эффект минимален.
3. LLDP flood (L2)

Текущая реализация:
Кадры Ether type 0x88cc с сырыми данными \x02\x07\x04\x00\x00\x00. Это лишь очень простой LLDP-пакет.

Предложения по улучшению:

    Формировать полноценный LLDPDU (LLDP Data Unit) с обязательными TLV: Chassis ID, Port ID, TTL, и опциональными (System Name, Description, Capabilities).

    Позволить настраивать Chassis ID (MAC, IP или строка) и Port ID.

    Увеличить размер пакета добавлением множества опциональных TLV или padding.

    Реализовать возможность отправки LLDP с истекшим TTL (TTL=0) для принудительного удаления соседа.

Применение:
Тестирование сетевого оборудования, поддерживающего LLDP (не только Cisco), на предмет уязвимостей в реализации протокола, а также для DoS-атак на системы мониторинга и управления.

Ожидаемый результат:
При интенсивной отправке может произойти переполнение таблиц LLDP-соседей, рост нагрузки на CPU, а в некоторых реализациях — крах процесса lldpd. Атака может быть использована для маскировки под легитимное устройство.
4. VLAN Hop (Double Tagging) (L2)

Текущая реализация:
Ether / Dot1Q(vlan=outer) / Dot1Q(vlan=inner) / IP / ICMP. Двойное тегирование отправляется на интерфейс.

Предложения по улучшению:

    Добавить поддержку произвольного протокола внутри (не только ICMP): TCP, UDP, ARP, а также сырой payload.

    Возможность указать MAC-адреса источника и назначения.

    Опция использовать случайные или фиксированные MAC.

    Добавить поддержку 802.1ad (QinQ с разными EtherType).

    Сделать настройку количества тегов (до 2, но можно и больше для тестирования аппаратных ограничений).

Применение:
Проверка уязвимости коммутаторов к двойному тегированию (CVE-2009-3539 и подобные). Позволяет отправлять пакеты в другой VLAN, обходя сегментацию, если коммутатор удаляет только один тег.

Ожидаемый результат:
При неправильной конфигурации коммутатора (или уязвимости) пакет попадёт в целевой VLAN (inner tag), что может привести к несанкционированному доступу к ресурсам другого сегмента.
5. ICMP Redirect (Network)

Текущая реализация:
ICMP type 5 (Redirect), code 1 (Redirect for host), с подменой шлюза. Отправляется на широковещательный MAC.

Предложения по улучшению:

    Возможность указывать код перенаправления (0 – для сети, 1 – для хоста).

    Отправлять пакет непосредственно жертве (unicast), а не на broadcast, чтобы повысить вероятность принятия.

    Добавить опцию включать в пакет оригинальный IP-пакет, который вызвал redirect (согласно RFC), чтобы он выглядел легитимнее.

    Реализовать разные скорости отправки (один раз, периодически, flood).

Применение:
Атака на хосты в локальной сети с целью изменения их маршрутизации (MITM). Может использоваться для перехвата трафика.

Ожидаемый результат:
Большинство современных ОС игнорируют ICMP Redirect, если пакет пришёл не от текущего шлюза, поэтому эффективность низкая. Однако в старых системах или при определённых настройках можно добиться перенаправления трафика.
6. Smurf (Network)

Текущая реализация:
ICMP Echo Request с source IP = жертва, destination IP = broadcast-адрес подсети.

Предложения по улучшению:

    Дать возможность задавать любой тип ICMP (не только echo).

    Поддержка IPv6 (ICMPv6).

    Добавить опцию использования directed broadcast (если маршрутизатор разрешает).

    Реализовать возможность указания количества повторных запросов или бесконечный режим.

    Добавить статистику (packets sent, pps).

Применение:
Классическая DoS-атака с усилением трафика. Жертва получает множество ответов от всех хостов в сети. Актуальна только в сетях, где разрешён broadcast forwarding.

Ожидаемый результат:
При разрешённом directed broadcast нагрузка на жертву и канал может быть значительной. В современных сетях broadcast обычно заблокирован на маршрутизаторах, поэтому атака может не работать.
7. Fraggle (Network)

Текущая реализация:
UDP-пакеты с source IP = жертва, destination IP = broadcast, порт назначения задаётся, source port = 7 (echo), payload = 'x'*64.

Предложения по улучшению:

    Сделать порт источника настраиваемым.

    Разрешить выбор протокола (UDP/TCP) и возможность сырого payload.

    Добавить поддержку направленного broadcast.

    Увеличить размер полезной нагрузки для усиления эффекта.

Применение:
Аналогично Smurf, но на основе UDP. Может быть эффективнее, если в сети есть сервисы, отвечающие на UDP-запросы (echo, chargen, qotd).

Ожидаемый результат:
Усиление трафика за счёт ответов от UDP-сервисов в сети. В современных сетях такие сервисы обычно отключены, поэтому эффект слабый, но возможно использование для сканирования.

        MAC Flooding: отправка кадров с разными source MAC на один порт для переполнения CAM-таблицы коммутатора.

        IGMP/MLD Query Flood: для нарушения работы multicast.
